# Auth, Location Fallback, And iOS 27 UI Design

Date: 2026-06-20
Project: `J:\电表统计`

## Context

The current application is a Cloudflare Worker with Workers Static Assets and D1. The Worker stores meter readings in the `readings` table and app settings in the existing `settings` table. `wrangler.toml` intentionally does not contain `database_id`; production D1 is bound in Cloudflare Dashboard with binding name `DB`.

The live site can load and read D1, but two product gaps remain:

- The structured Chinese address can fail Nominatim geocoding, leaving `settings.location` null and blocking weather.
- The UI still reads like the old dashboard and still depends on `unpkg.com` at runtime.
- There is no login barrier for private household readings and API keys.

The Caiyun Weather API uses `/v2.6/{token}/{longitude},{latitude}/weather`. The token is sensitive and must not be hard-coded, committed, or documented verbatim.

## Approved Scope

Implement the current request as one cohesive update:

- Add login with default user `admin` and initial password `password`.
- Force a password change after first login before allowing business API access.
- Protect all `/api/*` business routes; keep only auth endpoints public.
- Add browser geolocation and manual latitude/longitude fallback so weather is not blocked by failed structured-address geocoding.
- Rebuild the UI in an iOS 27 / Liquid Glass inspired style using local static assets.
- Preserve Cloudflare Worker + D1 deployment and Dashboard D1 binding.
- Update README and deploy documentation.

## Authentication Design

Static assets remain public so the login page can load. API routes are protected server-side.

Public auth endpoints:

- `GET /api/auth/status`: returns whether the current session is authenticated, whether password change is required, and the current username.
- `POST /api/auth/login`: accepts `{ username, password }`, verifies credentials, creates an HttpOnly session cookie, and returns `mustChangePassword`.
- `POST /api/auth/change-password`: requires a valid session, verifies `currentPassword`, stores a PBKDF2 password hash for `newPassword`, clears the forced-change flag, and returns a refreshed session.
- `POST /api/auth/logout`: removes the current session and clears the cookie.

Protected business endpoints:

- `/api/state`
- `/api/settings`
- `/api/settings/geocode`
- `/api/settings/location`
- `/api/weather`
- `/api/readings/*`
- `/api/prompt`

Auth data is stored in the existing D1 `settings` table under key `auth`, not in a new table. Passwords are never stored in plaintext. The Worker uses Web Crypto PBKDF2 with SHA-256, a per-password random salt, and session tokens generated by the Worker runtime. Session cookies are `HttpOnly`, `SameSite=Lax`, `Path=/`, and `Secure` on HTTPS.

If no auth row exists, the Worker accepts only `admin` / `password`, creates the first auth record, and sets `mustChangePassword: true`. While `mustChangePassword` is true, only auth status, change-password, and logout are allowed.

## Location Design

Structured address geocoding remains supported and still saves settings even if geocoding fails. Geocoding failure is treated as a recoverable warning.

Add a protected endpoint:

- `PUT /api/settings/location`

Request body:

```json
{
  "latitude": 22.6849,
  "longitude": 113.9426,
  "addressText": "广东省 深圳市 宝安区 石岩街道 新风路10号",
  "provider": "Browser Geolocation"
}
```

Validation:

- `latitude` must be between `-90` and `90`.
- `longitude` must be between `-180` and `180`.
- `addressText` and `provider` are trimmed strings.

The settings page adds:

- A current-location button using `navigator.geolocation.getCurrentPosition`.
- Manual latitude and longitude inputs.
- A save-coordinate button.
- A location summary that clearly shows whether weather can run.

Weather continues to call Caiyun on the Worker only. The browser never receives the plaintext Caiyun token.

## UI Design

The UI becomes a practical iOS 27 / Liquid Glass productivity console:

- A translucent login shell is the first screen until authenticated.
- A forced-password-change panel replaces the app while the default password is active.
- Desktop uses a glass side rail, glass toolbar, status strip, balanced overview, quick-entry panel, trend chart, metrics, and compact panels.
- Mobile uses a bottom glass navigation bar and stacked panels with 44px+ touch targets.
- The settings page combines structured address, weather token, browser geolocation, and manual coordinates in one workflow.
- External runtime assets are removed. Lucide is served from `public/vendor/lucide.min.js`.

The visual system avoids a one-color theme by combining neutral graphite text, green energy status, amber gas/warning accents, blue weather accents, and a soft CSS-only multi-color background. Cards keep radii under 8px where they represent repeated data items; glass panels and shells may use larger radii for the iOS-style material.

## Testing

Use Node's built-in `node:test`.

Worker tests cover:

- Unauthenticated business API returns `401`.
- Default `admin` / `password` login returns `mustChangePassword: true`.
- Sessions with forced password change receive `403` on business APIs.
- Changing password rejects invalid current passwords and weak/default new passwords.
- After password change, the old password fails and the new password works.
- A valid session can access `/api/state`.
- Manual location save persists `settings.location`.

Static tests cover:

- No `unpkg.com` or `@latest` runtime dependency.
- Local lucide vendor script is used.
- Login and change-password DOM elements exist.
- iOS glass shell selectors exist.
- Manual coordinate and browser geolocation controls exist.

Manual verification covers:

- Local Worker API and browser login flow.
- Desktop and mobile screenshots for non-overlapping UI.
- Cloudflare dry-run and production deploy.

## Deployment

No D1 schema migration is required. The existing `settings` table stores both app settings and auth state. The production Worker must keep a D1 binding named `DB`.

Deployment remains:

```powershell
npm run deploy
```

After deployment:

1. Open the public Worker URL.
2. Log in with `admin` / `password`.
3. Change the password immediately.
4. Enter the Caiyun token in Settings or keep the existing D1 value.
5. Use browser geolocation or manual coordinates if structured geocoding fails.

## Acceptance Criteria

- Login is required before any business API returns household data.
- First login forces a password change.
- Password hashes and session tokens are stored in D1; plaintext passwords are not.
- Structured-address failure no longer blocks weather forever because manual/browser location can save coordinates.
- The UI visibly matches the iOS-style glass console direction on desktop and mobile.
- The app no longer loads `unpkg.com` at runtime.
- `npm test` passes.
- `npx wrangler deploy --dry-run` passes.
- The app deploys to the configured Cloudflare Worker and runs with Dashboard D1 binding `DB`.
